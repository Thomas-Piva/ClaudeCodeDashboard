Design: Bugfix Batch v4.5.0

6 fix indipendenti — dettaglio di ogni modifica

1

claudeDirNameToPath — reverse conversion lossy

claude-watcher.js:469
Problema
C--Progetti-Pilota-MioProgetto → C:\Progetti\Pilota\MioProgetto
Spazi nel path originale (→ -) vengono convertiti in \
→
Fix
Sostituire la decodifica "indovina" con un rescan mirato delle scan roots quando appare una nuova dir in .claude/projects. Invece di ricostruire il path dal nome, si usa discoverFromRoots() già esistente — confronta il claudeDirName calcolato con quello trovato.
2

localhost:3001 hard-coded nel frontend

ProjectCard.jsx (4 punti)
Problema
fetch('http://localhost:3001/api/...')
Duplicato in mark-checked, exclude, open-terminal, terminal-windows, focus-window
→
Fix
Creare src/config.js: export const API_BASE = import.meta.env.VITE_API_BASE || 'http://localhost:3001' e sostituire tutte le occorrenze. Un solo punto da cambiare.
3

terminal-pids.json accumula voci stale

server.js:171
Problema
PID di processi morti rimangono nel file — al prossimo "trova finestra" viene restituito un match fantasma con badge "dashboard"
→
Fix (2 punti)
1. In /exclude: rimuove il PID del progetto escluso dal file.
2. In /terminal-windows: se il PID salvato non è presente nell'output PS (processo morto), rimuoverlo dal file in modo reattivo.
4

readLastNLines legge tutto il file

claude-watcher.js:90
Problema
Su sessioni lunghe (.jsonl da MB), readFileSync() carica tutto in RAM ogni 10s per ogni progetto
→
Fix
Leggere solo gli ultimi 32KB del file con fs.readSync(fd, buf, 0, 32768, fileSize - 32768) — sufficiente per 20+ righe JSONL. Stesso approccio anche per readLastLine().
5

alert() / confirm() nativi

ProjectCard.jsx:128,169
Problema
confirm("Escludere...") e alert("Errore") — dialoghi di sistema grigi, bloccanti, fuori tema
→
Fix
Per confirm: stato confirmingExclude — il bottone ⊗ diventa inline "CONFERMA / ANNULLA" per 3s.
Per alert (errori): piccolo testo rosso sotto il bottone che scompare dopo 3s. Zero dipendenze.
6

Nessun rate limit su /terminal-windows

server.js:232
Problema
Ogni chiamata avvia un processo PowerShell pesante (~150 righe). Nessun blocco se già in esecuzione per lo stesso progetto.
→
Fix
Map pendingTerminalRequests in server.js: se c'è già una richiesta in corso per quel progetto, risponde con 429 + messaggio. Il frontend mostra già lo stato loadingTerminal che disabilita il bottone — il fix backend è il secondo livello di protezione.
